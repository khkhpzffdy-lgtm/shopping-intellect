SLICE §2.8b — StoreProduct detail screen (name, photo, barcode)

BUDGET
  Target: one Codex/Claude session, under ~25 min, under ~200 changed lines.

GOAL
  Tapping a list item backed by a StoreProduct (a manually-recorded specific item, e.g.
  "Мляко Данон 2% 1л") opens a real detail screen — edit name, photo, barcode — instead
  of doing nothing (planning-mode rows with no `user_product_client_uuid` currently
  render as a plain, non-interactive `<div>` in `ListScreen.tsx`).

SCOPE
  - Backend: PATCH /store-products/{id} — name (re-run TermNormalizer, reject blank),
    image_url, and barcode_value (writes to the existing, currently-untouched
    `oCk_si_barcodes` table — no PHP code touches that table yet at all: no model,
    no repository, no controller).
  - Backend: new BarcodeRepositoryInterface + WpdbBarcodeRepository (attach a barcode
    value to a store_product_id; list barcode values for one; replace — delete any
    existing values for that store_product_id, then insert the new one). The detail
    screen's single barcode field always calls `replace()`, never `attach()` directly
    — `attach()` stays on the interface for Phase-2 multipack/variant support, which
    is out of scope here. See "Resolved — barcode edit is a true replace, not an
    accumulate (2026-06-21)" in decisions.md.
  - Backend: new StoreProductController (none exists yet — GET/PATCH /store-products/{id}
    don't exist; ListController only ever *creates* or *reads via list item* a
    StoreProduct, never fetches/edits one standalone).
  - Frontend: new StoreProductDetailScreen.tsx, opened from ListScreen.tsx's
    planning-mode item row when `item.store_product_client_uuid` is set (the row
    currently renders as inert `<div className="git__main">` — mirror the exact
    `<button>`-vs-`<div>` branch already added for UserProduct-backed rows in §2.8a,
    just on the other field).
  - Frontend: wire the existing-but-dead-end `manualBarcode` field in
    AddSearchScreen.tsx's manual-creation form (`app/src/components/AddSearchScreen.tsx`,
    state already exists, currently typed into but never sent anywhere — confirmed by
    reading `addManualStoreProduct()`, which builds the `StoreProduct` object without it)
    to actually persist via the new barcode endpoint once the product is created.

OUT OF SCOPE
  - UserProduct detail (already done, §2.8a).
  - Multiple barcodes per product, barcode scanning (camera), barcode-driven
    cross-store bucketing (Phase 2, `decisions.md` still has this as an open question —
    don't build matching logic here, just store the value).
  - §4.0d (Catalog bucket detail) — separate slice, comes after this one.
  - Family-shared edit rights on a StoreProduct — deferred until family-owned lists
    actually exist (they don't yet — see RESOLVED below). Don't build a family-
    membership check into StoreProductService ahead of that.

RESOLVED — StoreProduct edit rights (was an open question, now settled, 2026-06-21)
  Edit rights on a `source='user'` StoreProduct are **creator-only**, checked against
  `created_by_user_id` — same shape as `UserProductService::ownedNonSystemUserProduct()`'s
  guard. A non-creator's edit attempt returns **403 Forbidden** (a new
  `StoreProductForbiddenException`), mirroring the existing system-row 403 in
  `UserProductController` — not a silent no-op, not a fork into a new row.

  Why creator-only and not "anyone on a shared family list": `StoreProduct` dedupe is
  global across all users (`findByNormalizedName()` has no owner filter — confirmed),
  so two unrelated users can already share one row. "Family members can edit" would
  need a family-membership check, but family-owned lists aren't built yet —
  `ListService::createList()` rejects any `owner_type` other than `'user'`
  (`ListService.php:39`) and `ownsList()` only checks `ownerType === 'user' && ownerId
  === userId` (`ListService.php:268`). `family_ids[]` exists on the JWT claim set but
  nothing in `ListService` uses it yet. Don't build family-aware authorization ahead of
  that feature existing — see decisions.md for the full resolution.

RELEVANT CANON (quoted)
  - `13-implementation-line.md` §2.8: "StoreProduct detail: name (editable), optional
    photo (`image_url`), optional barcode (writes to `oCk_si_barcodes`), and — only for
    a `source='user'` row — clear affordance that this is a manually-recorded item, not
    yet matched to any store."
  - `04-database.md` §4.x, `oCk_si_barcodes`: "A product may carry several (multipack/
    variant); one barcode value recurs across stores (which is exactly what lets Phase
    2 pull offers into one bucket)." `UNIQUE (store_product_id, barcode_value)`. FK
    `store_product_id` → `store_products.id` `ON DELETE CASCADE`.
  - `04-database.md` `oCk_si_store_products`: `source` ENUM('crawler','user'),
    `created_by_user_id` nullable FK, `normalized_name`, `image_url` nullable,
    `is_archived`.
  - Existing pattern to mirror: `UserProductService::rename()`/`ownedNonSystemUserProduct()`
    guard shape (`src/Services/UserProductService.php`) and
    `UserProductController::handlePatch()` (`src/Api/UserProductController.php`) — same
    400/403/409 error-mapping conventions, same "diff and apply only what changed" style
    used by `setCategories()`. The 403-on-non-owner behavior for `UserProductService` is
    an implementation detail recorded in `STATUS.md` (search "403 on a system row"), not
    a named `decisions.md` resolution — there is no `decisions.md` entry literally
    titled "favorites on a system row"; don't search for one.

IRON RULES IN PLAY (CLAUDE.md §2)
  - Prepared statements only; `Repositories/Wpdb` is the only place touching `$wpdb` —
    the new `WpdbBarcodeRepository` follows the exact same shape as
    `WpdbUserProductRepository`.
  - "A `list_items` row targets exactly one of `user_product_id` or `store_product_id`"
    — this slice never changes that; it only edits the StoreProduct a row already
    points at.
  - `StoreProduct` carries `source` (`crawler`|`user`) — a `source='crawler'` row (a
    real crawled offer, once §3.1+ crawlers exist) must **never** be editable by an
    ordinary user the same way a `source='user'` row is — lock name/photo/barcode
    editing to `source === 'user'` rows only, mirroring the system-row lock pattern from
    §2.8a (disable, don't hide, so the Owner can still see it's intentional).
