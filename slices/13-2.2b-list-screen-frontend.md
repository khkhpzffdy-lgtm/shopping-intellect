GOAL
  Replace the static "No lists yet" `HomeScreen` placeholder with a real Lists
  overview and a two-mode (planning/shopping) List screen. Creating a list and
  adding/checking/removing items applies **optimistically** to local state and
  persists across reload — including while offline. This slice writes through an
  IndexedDB mirror and an outbound mutation log; **actually flushing that log to the
  server on reconnect is §2.3** (this slice may do a simple best-effort online POST
  alongside the optimistic local write, but the durable background-sync/retry engine
  is out of scope — see OUT OF SCOPE).

SCOPE
  - **IndexedDB wrapper** (`src/storage/db.ts`, use `idb` — add as a dependency):
    object stores `lists`, `list_items`, `mutation_queue`, per `07 §4.1`/§5.1. Keyed
    by `client_uuid`.
  - **Lists overview** (`src/components/ListsScreen.tsx`, replaces the "No lists yet"
    block in `HomeScreen`): reads lists from IndexedDB (instant render), shows
    `ListCard`s with name + item count, and the existing "Create list" button now
    opens a name-input flow (per Flow 2): on confirm, write a new list row to
    IndexedDB with a generated `client_uuid`, `owner_type: 'user'`, optimistically
    show it, queue a `POST /lists` mutation, and best-effort `apiRequest` it now if
    `navigator.onLine`.
  - **List screen** (`src/components/ListScreen.tsx`): opens a list by `client_uuid`
    or server `id`, shows its items from IndexedDB. Two-mode via `ModeToggle`
    (planning/shopping — `10 §1.5`/§2.6, remembered per list in a small Zustand store
    `src/store/listMode.ts`, e.g. `Record<listId, 'planning'|'shopping'>`):
    - **Planning mode** — rows are expandable (expand can be a no-op/stub this
      slice — Product Detail is §2.x later), show qty/unit, no price.
    - **Shopping mode** — large checklist rows, tap toggles `is_checked`, no price,
      not expandable.
    - Both modes: add item via a simple text input (term) + qty/unit fields → writes
      a new `list_items` row (and, if the term doesn't match an existing local
      `user_products` entry, a new `user_products` row) to IndexedDB with
      `client_uuid`s, optimistic render, queue `POST /lists/{id}/items` with inline
      `user_product` per `06 §6.2`, best-effort send if online.
    - Remove a line → optimistic local delete + queue `DELETE .../items/{itemId}`.
  - **`SyncStatusIndicator`** (`10 §8.19`, minimal): a small badge per list/item
    showing "synced" / "sync-pending" (count of pending mutations for that entity) —
    no need for "sync-failed" handling yet (that's §2.3's retry logic).
  - **Routing**: simplest viable — e.g. local state in `App.tsx` switching between
    `ListsScreen` and `ListScreen(listId)`, no router library needed unless one's
    already present (check `package.json` first).
  - Tests (Vitest + Testing Library, following `src/test/App.test.tsx` patterns):
    creating a list shows it immediately; adding an item shows it immediately;
    reloading (re-mounting, simulating IndexedDB persistence) still shows the list
    and item.

OUT OF SCOPE
  - Service Worker / Background Sync registration, exponential backoff, retry on
    `5xx`, `409 duplicate_client_uuid` reconciliation — all §2.3. This slice's
    "best-effort online POST" may simply fail silently and leave the mutation in
    `mutation_queue` with `status: 'pending'`; §2.3 builds the flush engine that
    drains it.
  - Product Detail / candidate browsing (Flow 4), Comparison (Flow 6), brand
    anchoring (Flow 5), favorites/recent/frequent quick-add (Flow 10) — all later
    slices.
  - Family lists / `OwnershipBadge` family variant — §2.4.
  - `AttributionChip` (`added_by` on family lists) — §2.4 (personal lists only here).

  - No merge-UX, no conflict dialogs (D §14, `10 §7.6`) — not relevant yet since §2.3
