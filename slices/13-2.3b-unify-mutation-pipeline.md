SLICE <13-§2.3b> — Unify the mutation pipeline: one `sendMutation`, no per-screen copies

BUDGET
  Target: one Codex session, under ~35 min, under ~200 changed lines (excluding test
  file diffs). This is a **refactor for correctness**, not a feature — behaviour must
  stay identical for every existing acceptance criterion of §2.2/§2.3/§4.0. If you find
  yourself changing what a mutation does rather than how it's dispatched, STOP and
  hand back per escalation §5 (type 3).

GOAL
  Every optimistic write in the app — present (list, item, user-product, checked
  toggle, item removal) and future (favorites, brand anchors, profiles, anything
  later) — sends through **one shared function**, identical for the immediate
  "try now while online" attempt and the background queue flush. Today there are
  **five independent hand-written copies** of "enqueue, then immediately also try to
  send it, then apply success or leave it queued" — which is *why* §2.3a's bug (and
  the auth bugs before it) could appear fixed for "create a list" while "add an item"
  stayed broken: they were never the same code.

SCOPE
  1. **Create `app/src/sync/sendMutation.ts`** exporting an async function
     `sendMutation(mutation: MutationQueueRecord): Promise<void>` that:
     - Resolves the real endpoint via the existing `resolveEndpoint` logic currently
       private to `app/src/sync/flush.ts` (move/export it, don't duplicate it).
     - Calls `apiRequest` with the mutation's method/body/`authenticated: true`.
     - On success: calls `applyMutationSuccess(mutation, response)` then
       `markMutationDone(mutation.client_uuid)`.
     - On failure: calls `markMutationFailed(mutation.client_uuid)` and rethrows, so
       the caller can decide what (if anything) to show the user — but the queue
       state is **always** correctly updated by `sendMutation` itself, never left to
       each caller to remember.
  2. **Refactor `app/src/sync/flush.ts`'s `drainQueuedMutations`** to call
     `markMutationInFlight` then `sendMutation(claimedMutation)` per queued record,
     replacing its current inline `apiRequest`/`applyMutationSuccess`/
     `markMutationDone`/`markMutationFailed` block with a call to the new shared
     function. Behaviour (claim-then-send, continue on a single failure, return the
     same `FlushResult` shape) must not change.
  3. **Refactor `app/src/components/HomeScreen.tsx`** — all four call sites:
     - `handleCreateList` (~line 153-225): after the optimistic `putList` +
       `enqueueMutation`, call `markMutationInFlight` then `sendMutation` instead of
       its own inline `apiRequest` + `applyMutationSuccess` + `markMutationDone`
       block. Keep the existing `try/catch` around it for the user-facing
       `formatActionError` message — `sendMutation` already guarantees the queue
       state is correct even if this catch only shows text.
     - `handleAddItem` (~line 227-336): same treatment. **Remove the early
       `if (!selectedList.id) { return; }` silent bail-out (~line 287-289)** — it
       currently skips sending entirely (and skips showing any error) whenever the
       parent list hasn't synced yet; `sendMutation`/`resolveEndpoint` already knows
       how to wait for/resolve the parent list's real id, so this special case is no
       longer needed and was actively hiding failures from the user.
     - `handleToggleChecked` (~line 338-391): same treatment.
     - `handleRemoveItem` (~line 393-...): same treatment.
  4. **Refactor `app/src/components/AddSearchScreen.tsx`** (~line 114 area): same
     treatment for its inline add-item mutation.
