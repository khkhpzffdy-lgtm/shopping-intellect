SLICE <13-§1.4b> — Google OAuth login (`POST /auth/google`)

BUDGET
  Target: one Codex session, under ~20 min, under ~150 changed lines (excluding
  generated lockfiles). If you're past this and still not green, STOP — do not keep
  iterating. Report what's done, what's failing, and hand back per escalation §5
  (type 3) rather than burning more time/tokens.

GOAL
  A user can authenticate with their Google account and get back the exact same
  session shape (access JWT + refresh cookie + user object) as email/password
  login — so the frontend's session handling needs zero provider-specific code.

SCOPE
  - New REST endpoint `POST /auth/google` in `AuthController`, registered alongside
    the existing `/auth/login` etc.
  - New `GoogleAuthVerifierInterface` (in `Repositories/Contracts`) with one method
    that takes a Google authorization code + redirect URI and returns a verified
    `{ email, display_name }` pair (or throws `GoogleVerificationFailedException`).
  - A `GoogleAuthVerifier` implementation (in `Repositories/Wp` or a new
    `Repositories/Google` namespace — match the existing `Repositories/Wp` pattern)
    that calls Google's token + userinfo endpoints using `wp_remote_post` /
    `wp_remote_get` (WP core HTTP functions are fine here — this is identity, not
    a crawler; crawlers' `HttpClient` interface is a separate, not-yet-built seam).
  - `AuthService::loginWithGoogle(string $email, ?string $displayName, array $familyIds = []): AuthSession`
    — find-or-create the `wp_users` record via `AuthProviderInterface` (reuse
    `findByEmail` + `createUser`; Google-created users get a random unguessable
    password since `wp_insert_user` requires one) and issue a session exactly like
    `register`/`login`.
  - Config: Google OAuth client ID/secret read via `ConfigInterface` — add
    `googleClientId(): string` and `googleClientSecret(): string` to
    `ConfigInterface`/`Config`, sourced from WP options
    (`get_option('si_google_client_id')` etc.) or constants — Codex's choice,
    document whichever in STATUS.md.
  - PHPUnit tests for `AuthService::loginWithGoogle` (find-vs-create paths) and for
    `AuthController::handleGoogle` (success + `google_verification_failed`), with
    `GoogleAuthVerifierInterface` mocked — no real network calls in tests.

OUT OF SCOPE
  - The frontend "Sign in with Google" button / OAuth redirect flow (that's §1.5).
  - Apple/Facebook or any other provider.
  - Storing the Google `sub`/provider id (Stage 1 matches by email only — if this
    needs revisiting, flag it as an open question rather than inventing a new
    column).
  - Building the general-purpose crawler `HttpClient` interface — this slice's
    Google verifier is a narrow, identity-only seam.

RELEVANT CANON  (quoted, not just referenced — Codex does not load 00–11)
  - `06-api-auth.md` §6.1 (endpoint spec):
    ```
    POST /auth/google — Public — Google authorization code → verify with Google →
    find-or-create the wp_users record → identical output to /auth/login.

    → { "code": "<google_auth_code>", "redirect_uri": "https://app.<domain>/..." }
    ← 200  { "auth": {...}, "user": {...} }   + Set-Cookie: si_refresh=...
    errors: 401 google_verification_failed · 429 rate_limited
    ```
  - `06-api-auth.md` §7.2: "Both POST /auth/login and POST /auth/google terminate
    in the identical { auth: {...}, user: {...} } shape ... so the frontend cannot
    tell providers apart."
  - `decisions.md` §8 / D §14: identity stays behind `AuthProvider`/`UserRepository`
    over `wp_users`; provider abstraction must not leak into the JWT contract or
    the response shape.

IRON RULES IN PLAY  (from CLAUDE.md §2 — breaking one is a bug)
  - Business logic (AuthService) stays plain PHP behind constructor-injected
    interfaces — no direct WP calls from AuthService; only the Wp/Google
    repository implementations touch WP core functions.
  - Access JWT in memory only (returned in body); refresh token in an `httpOnly
    Secure` cookie — reuse `AuthController`'s existing cookie helpers verbatim,
    do not duplicate or alter the cookie logic.
  - Exact-origin CORS — do not add new CORS handling; this endpoint rides the
    existing `si/v1` route registration and CORS setup.

BUILD INSTRUCTIONS  (this is the prompt Codex acts on)
  1. Add `GoogleAuthVerifierInterface` to `src/Repositories/Contracts/`:
     `verify(string $code, string $redirectUri): GoogleIdentity` returning a small
     value object/array with `email` and `displayName` (nullable), throwing
     `GoogleVerificationFailedException` (new, extends `RuntimeException`, same
     pattern as `DuplicateEmailException`) on any failure (bad code, network error,
     unverified email).
  2. Implement `GoogleAuthVerifier` (pick `Repositories/Wp` or new
     `Repositories/Google` dir — be consistent with existing namespace
     conventions) using `wp_remote_post` to exchange the code at Google's token
     endpoint and `wp_remote_get` to fetch the userinfo, extracting `email` and
     `name`. Inject `ConfigInterface` for client id/secret.
  3. Extend `ConfigInterface` + `Config` with `googleClientId(): string` and
     `googleClientSecret(): string`, backed by WP options
     `si_google_client_id` / `si_google_client_secret` (document the option names
     in STATUS.md so the Owner knows what to set in wp-admin later).
  4. Add `AuthService::loginWithGoogle(string $email, ?string $displayName, array $familyIds = []): AuthSession`:
     - `findByEmail($email)`; if null, `createUser($email, <random 32-byte token>, $displayName)`.
     - Then call the same session-issuing path as `register`/`login`
       (`issueSession` — make it accessible if it's currently `private`, or factor
       a small shared helper; do not duplicate the refresh-token/JWT logic).
  5. Wire `AuthController`:
     - Register `POST /auth/google` -> `handleGoogle`, `permission_callback => '__return_true'`.
     - `handleGoogle` reads `code` and `redirect_uri` from the request body, calls
       the injected `GoogleAuthVerifierInterface::verify(...)`, catches
       `GoogleVerificationFailedException` -> `401 google_verification_failed`
       ("Unable to verify Google account."), otherwise calls
       `AuthService::loginWithGoogle(...)` and returns `sessionResponse($session)`
       (the existing private helper — reuse it unchanged).
  6. Update the plugin's DI/bootstrap wiring (wherever `AuthController`/`AuthService`
     are constructed) to supply the new `GoogleAuthVerifierInterface` binding.
  7. Write and green the PHPUnit tests listed below using the existing
     `wp-stubs.php` fakes — mock `GoogleAuthVerifierInterface` directly, do not
     stub `wp_remote_*` unless `GoogleAuthVerifier` itself is under test.

TESTS  (Codex writes and greens these)
  - `AuthService::loginWithGoogle` creates a new `si_user` (with `si_user` role)
    when no existing user matches the email, and returns a valid `AuthSession`
    (access token, refresh token, userId, displayName).
  - `AuthService::loginWithGoogle` reuses an existing user (no duplicate created)
    when the email already exists.
  - `AuthController::handleGoogle` returns 200 with the same `{ auth, user }` shape
    as `handleLogin`/`handleRegister`, and sets the `si_refresh` cookie, on a
    verifier success.
  - `AuthController::handleGoogle` returns `401 google_verification_failed` when
    the verifier throws `GoogleVerificationFailedException`.
