GOAL
  Writing a term (e.g. "мляко") creates or reuses a `UserProduct` for the list's
  owner, deduped on `(owner_type, owner_id, normalized_term)`. This is the
  demand-first birth point (D §4/§9): `list_items` reference `user_product_id`,
  never a free-text term or a canonical product directly.

SCOPE
  - New `TermNormalizer` (in `Support/`) implementing the resolved normalizer
    (`decisions.md` §14, `04 §7.1`): NFC + lowercase + trim + whitespace-collapse +
    punctuation-strip. No stemming on the dedup key.
  - New `UserProductService` (in `Services/`) with
    `findOrCreate(string $ownerType, int $ownerId, string $term, ?string $clientUuid = null): UserProduct`:
    - Normalize `$term` via `TermNormalizer`.
    - `findByOwnerAndNormalizedTerm($ownerType, $ownerId, $normalizedTerm)` — if found,
      return it (un-archiving via `update()` if `isArchived` was true — 04 §4.3).
    - Otherwise `insert()` a new `UserProduct` (`categoryId: null`, `brandAnchor: null`,
      `isFavorite: false`, `isArchived: false`, `clientUuid: $clientUuid ?? <new uuidv4>`).
    - Catch `DuplicateUserProductTermException` / `DuplicateClientUuidException` from
      the repository on a race — re-fetch via `findByOwnerAndNormalizedTerm` and return
      the existing row (idempotent).
  - New REST endpoints in `AuthController`-sibling controller (new `UserProductController`
    in `Api/`), registered in `RestApiBootstrap`:
    - `GET /user-products` — JWT required. Query params `owner_type`, `owner_id`
      (default to the caller's own `user_id`/`owner_type=user` if omitted — 06 §7.1).
      Returns `{ user_products: [...] }` per `06 §7.1` shape (id, term, normalized_term,
      category_id, brand_anchor, is_favorite — `last_purchased_at` /
      `purchase_count_window` are OUT OF SCOPE for this slice, omit or null them;
      `purchase_log` doesn't exist yet).
    - `POST /user-products` — JWT required. Body `{ client_uuid, term, owner_type, owner_id }`.
      Calls `UserProductService::findOrCreate`. Returns `201 { user_product: {...} }`
      per `06 §7.2` shape.
  - PHPUnit tests for `TermNormalizer` (NFC/lowercase/trim/collapse/punctuation-strip
    cases incl. Cyrillic) and `UserProductService::findOrCreate` (new-term path,
    repeat-term dedup path, re-archive path), using `tests/Repositories/SqliteWpdb.php`
    + `WpdbUserProductRepository` (already implemented).

OUT OF SCOPE
  - `list_items` / `POST /lists/{id}/items` inline `user_product` creation (06 §6.3) —
    that's a later slice; this slice only builds the standalone `UserProductService`
    and `/user-products` endpoints it will be reused by.
  - `purchase_log`, favorites toggling, brand anchoring, categorization — all later.
  - Family-membership authorization checks beyond "caller is JWT-authenticated" — if
    `owner_type=family`, do NOT yet verify family membership (flag as a TODO comment
    only if trivial; do not build the family_members check here).

IRON RULES IN PLAY (from CLAUDE.md §2 — breaking one is a bug)
  - Business logic (`UserProductService`, `TermNormalizer`) stays plain PHP behind
    constructor-injected interfaces — only `Repositories/Wpdb/WpdbUserProductRepository`
    (already implemented) touches `$wpdb`.
  - `(owner_type, owner_id, normalized_term)` uniqueness is enforced by the repository
    (`DuplicateUserProductTermException`) — the service must handle the race, not
    pre-check-then-insert.
  - `client_uuid` (UUIDv4) required on every `user_products` row (§2.6).
  - Exact-origin CORS / JWT auth — reuse `AuthController`'s existing JWT-verification
    middleware/helper for the new controller, do not duplicate or alter it.

TESTS (Codex writes and greens these)
  - `TermNormalizer::normalize`: " Мляко  3.2%! " -> "мляко 3.2" (or equivalent
    punctuation-stripped form — Codex documents the exact expected output for the
    Cyrillic + punctuation + multi-space case in a test comment), plus a plain
    ASCII case and a pure-whitespace-collapse case.
  - `UserProductService::findOrCreate`: new term for an owner creates one
    `UserProduct` row with `category_id: null`, `is_archived: false`, a generated
    `client_uuid`.
  - `UserProductService::findOrCreate`: calling it twice with terms that normalize to
    the same value for the same owner returns the **same** `UserProduct` (same id),
    not two rows.
  - `UserProductService::findOrCreate`: if the existing matching row has
    `is_archived: true`, it is un-archived and returned (not duplicated).
  - `UserProductController`: `POST /user-products` returns `201` with the
    `{ user_product: {...} }` shape; `GET /user-products` returns `200` with
    `{ user_products: [...] }` for the caller's own terms.
