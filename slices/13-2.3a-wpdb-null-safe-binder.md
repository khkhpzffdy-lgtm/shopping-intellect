SLICE <13-§2.3a> — Wpdb null-safe value binder (fixes the "new term 500s" crash)

BUDGET
  Target: one Codex session, under ~25 min, under ~150 changed lines (excluding test
  file diffs). This is a **hardening/bugfix** slice — one shared helper plus three
  repositories that adopt it. If scope creeps into redesigning the repository layer,
  STOP and hand back per escalation §5 (type 3).

GOAL
  Adding a never-before-seen term to a list (the single most common action in the app)
  must not 500. No `Repositories/Wpdb` class is allowed to silently write `0`/`''` to
  the database for a PHP `null` value ever again.

SCOPE
  1. **Add one shared null-safe binder** in `plugin/src/Repositories/Wpdb/` (e.g. a
     final class `WpdbNullSafe` with a static method, signature roughly
     `bind(object $wpdb, int|string|null $value, string $type): string` where
     `$type` is `'d'` or `'s'`). It returns the literal SQL string `'NULL'` when
     `$value === null`, otherwise `$wpdb->prepare("%{$type}", $value)`. This is the
     standard workaround for `$wpdb->prepare()`'s documented behaviour: a `%d`
     placeholder casts `null` to `0`, a `%s` placeholder casts `null` to `''` — it can
     never emit SQL `NULL` itself.
  2. **Replace the stopgap in `WpdbUserProductRepository`** (`insert()`, `update()`):
     a minimal inline fix (a `$value === null ? 'NULL' : $wpdb->prepare(...)` ternary,
     marked `TODO(13-§2.3a)` in a comment) already shipped live on 2026-06-17 to
     unblock production — replace those two inline ternaries with calls to the new
     shared `WpdbNullSafe::bind()` helper instead of leaving them as one-off
     duplicated logic. (`category_id: null` is the **normal, expected** state for a
     brand-new term — D §4, "matching in progress" — not an edge case.)
  3. **Fix `WpdbUserProfileRepository::upsert()`** (line ~40): same treatment for
     `display_name` and `onboarding_state`.
  4. **Fix `WpdbRefreshTokenRepository::insert()`** (line ~25): same treatment for
     `rotated_at` and `revoked_at` (both always `null` at insert time — a fresh token
     has never been rotated/revoked — so this is a silent-corruption bug today, not
     yet a crash, because no FK references those columns).
  5. Confirm (don't need to fix — already clean) that `WpdbListRepository` and
     `WpdbListItemRepository` have no nullable-column writes to worry about.

OUT OF SCOPE (explicitly — do not pull these in)
  - The frontend mutation pipeline (duplicate `enqueueMutation` call sites) — that is
    `13-§2.3b`, a separate slice, intentionally sequenced after this one.
  - Any change to `Repositories/Contracts` interfaces, `Models/*` constructors, or
    REST response shapes — this slice only changes *how a value is bound into SQL*,

  - Building a query builder / ORM. One small static helper, not an abstraction layer.
  - Touching `WpdbListRepository`/`WpdbListItemRepository` beyond confirming (step 5)
    they're unaffected — don't refactor working code with no nullable columns.

IRON RULES IN PLAY (from CLAUDE.md §2 — breaking one is a bug)
  - "Only `Repositories/Wpdb` may touch `$wpdb`." — the binder lives inside that
    directory, used only by repositories, never called from `Services`/`Api`.
  - Prepared statements exclusively — the binder still routes every non-null value
    through `$wpdb->prepare()`; it never string-concatenates a raw user value.
  - Categorization is lenient and non-blocking (`category_id: null` is a normal,
    expected, common state) — this slice makes that state actually work, it doesn't
    change when `null` is produced.
