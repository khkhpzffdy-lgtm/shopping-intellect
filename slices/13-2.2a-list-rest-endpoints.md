GOAL
  Provide the REST surface the frontend's two-mode list screen (§2.2b) needs:
  create/list/open lists, and add/edit/check/remove items on a list — referencing
  `user_product_id` only, with inline UserProduct creation for new terms (the
  demand-first birth point, D §4/§9).

SCOPE
  - New `ListController` (in `Api/`), registered in `RestApiBootstrap`, JWT-protected
    (reuse `AuthController`'s JWT-verification helper verbatim):
    - `GET /lists` — all lists the caller owns (personal only this slice — `family_ids[]`
      from the JWT claim exists but family-owned lists are OUT OF SCOPE, see below).
      Returns `{ lists: [{ id, name, owner_type, owner_id, item_count, updated_at }] }`
      per `06 §6.2`.
    - `POST /lists` — body `{ name, owner_type, owner_id, client_uuid }`. This slice
      only supports `owner_type: "user"` with `owner_id == caller's user_id`
      (family lists are §2.4 — if `owner_type: "family"` is sent, return
      `403 not_a_member` per the existing error vocabulary, do not implement family
      membership checks). Idempotent on `client_uuid` (`409 duplicate_client_uuid` →
      return existing list, not an error — `06 §6.2`). Returns `201 { list: {...} }`.
    - `GET /lists/{id}` — the list + its items, each item carrying its resolved
      UserProduct summary (`term`, `category_id`, `brand_anchor`) per `06 §6.2`'s
      shape — join via `UserProductRepositoryInterface::find()`. `404 not_found` if
      the list doesn't exist or isn't owned by the caller (covers forbidden, §4.3).
    - `POST /lists/{id}/items` — body per `06 §6.2`: either `user_product_id` (existing
      term) OR inline `user_product: { client_uuid, term }` (new term — call
      `UserProductService::findOrCreate` from §2.1, owner = the list's owner). Returns
      `201 { item: {...}, user_product: {...} }` (user_product always included, even
      when pre-existing — frontend reconciles either way). `409 duplicate_client_uuid`
      on item replay → return existing item. `404 list_not_found`.
    - `PATCH /lists/{id}/items/{itemId}` — body `{ is_checked }` or `{ quantity }` or
      `{ unit }`. Returns `200 { item: {...} }` with updated `updated_at`.
      `purchase_log` append on false→true `is_checked` is OUT OF SCOPE (no
      `purchase_log` table/repo exists yet — §2.5 territory; skip it, do not stub a
      table write).
    - `DELETE /lists/{id}/items/{itemId}` — removes the line (item row only, never
      touches the `user_product`). Returns `204`.
  - Reuse the existing `ListRepositoryInterface`/`WpdbListRepository`,
    `ListItemRepositoryInterface`/`WpdbListItemRepository`, `ShoppingList`/`ListItem`
    models — these already exist (check `src/Repositories/Wpdb/WpdbListRepository.php`
    and `WpdbListItemRepository.php`). Add a `ListService` (in `Services/`) only if
    the controller logic needs more than thin pass-through (e.g. the inline
    user_product birth + ownership check) — keep business logic out of the controller
    per CLAUDE.md §2 rule 1.
  - PHPUnit tests for `ListController`: create list (+ duplicate_client_uuid replay),
    get list with items, add item with inline user_product (asserts a UserProduct is
    created via `UserProductService`), add item with existing `user_product_id`, patch
    item (`is_checked`/`quantity`), delete item, and the `404`/`403` cases above.

TESTS (Codex writes and greens these)
  - `POST /lists` creates a list owned by the caller; replaying with the same
    `client_uuid` returns the same list id, not a new row.
  - `POST /lists` with `owner_type: "family"` returns `403 not_a_member` (no family
    membership table check needed — just reject family lists this slice).
  - `GET /lists` returns only the caller's own lists.
  - `GET /lists/{id}` returns `404` for another user's list; returns items with
    resolved `term`/`category_id`/`brand_anchor` for the caller's own list.
  - `POST /lists/{id}/items` with inline `user_product: { client_uuid, term: "прах
    Ariel" }` creates both a `UserProduct` (via `UserProductService::findOrCreate`)
    and a `list_item` referencing it.
  - `POST /lists/{id}/items` with an existing `user_product_id` creates only the item.
  - `PATCH /lists/{id}/items/{itemId}` with `{ "is_checked": true }` updates the item
    and `updated_at`.
  - `DELETE /lists/{id}/items/{itemId}` removes the item but leaves the
    `user_product` row intact.
