GOAL
  §2.2c restyled the Lists overview and List screen to match `design/screens2.jsx`, and §4.0
  added the Add/Search screen + bottom nav. Between the two, several **inconsistencies and one
  dead prop/button** crept in that directly contradict CLAUDE.md's "buttons should be the same
  and in the same place" expectation and this codebase's own stated rule (§2.2c slice text:
  "if the rest of the app is English, keep English copy... don't do a partial bilingual mix").
  This slice fixes the four concrete, low-risk issues below — no new screens, no new backend
  calls, no change to data flow.

  Owner-visible outcome: every screen reads in **one language**, the sync indicator and empty
  state look identical everywhere they appear (because they're the same component, not copies),
  and every list has one consistent, working way to reach the dedicated Add/Search surface.

SCOPE — four fixes, each independently verifiable

  1. **Dedupe `SyncStatusIndicator`.**
     `app/src/components/ListScreen.tsx` (lines ~22-27) and `app/src/components/ListsScreen.tsx`
     (lines ~18-23) each define an **identical** inline `SyncStatusIndicator` component. Extract
     one shared component (e.g. `app/src/components/SyncStatusIndicator.tsx`, matching the name
     already used in `10-ux-rules.md §8.19`) and import it in both places. No behaviour change —
     same markup, same classes (`listcard__badge`, `sync__dot`).

  2. **Wire the real `EmptyState` component instead of a duplicate inline copy.**
     `app/src/components/EmptyState.tsx` exists (`context: 'no-lists'`) but `ListsScreen.tsx`
     (lines ~142-152) renders its **own** inline empty-state markup instead of importing it —
     two near-identical empty states that will drift out of sync the next time either is edited.
     Additionally, `EmptyState.tsx`'s "Create list" button (line ~25-31) has **no `onClick`** —
     it is currently a dead button. Fix both: give `EmptyState` an `onCreate` (or similar) prop,
     call it from `ListsScreen.tsx`'s create-list handler, and have `ListsScreen.tsx` render
     `<EmptyState>` instead of its inline duplicate when `lists.length === 0`.

  3. **One language, not a partial mix.** Per the design canon (`design/screens2.jsx`,
     `design/README.md`) and this codebase's own newer screens (`ListsScreen.tsx`,
     `AddSearchScreen.tsx`), the app's user-visible copy is **Bulgarian**. The following are
     currently English and must be translated to match:
     - `ListScreen.tsx`: "This list is empty for now", the `Remove` button label (both planning
       and shopping row variants), the `piece`/`Item term`/`Item quantity`/`Item unit`
       *placeholders* (not the `aria-label`s — see note below).
     - `EmptyState.tsx`: "Lists overview" eyebrow, "No lists yet" heading, the body copy, "Create
       list" button label.
     - `HomeScreen.tsx`'s `formatActionError` fallback strings ("Could not create the list on
       this device yet...", "Could not add the item on this device yet...").
     - **Do not** change `aria-label`s or anything `app/src/test/*.test.tsx` queries via
       `getByLabelText`/`getByRole(..., { name: ... })` (e.g. `'Item term'`, `'Add item'`,
       `'List name'`) — those are accessibility/test selectors, not user-facing copy, and
       changing them would break the existing §2.2b/§2.2c Vitest suite for no visual gain.
       If a string is *both* a label and a selector, leave the selector value alone and only
       change visible text/placeholder content.

  4. **Fix the dead `onOpenAddSearch` prop with the design's own appbar search icon.**
     `ListScreen.tsx` accepts an `onOpenAddSearch` prop (passed down from `HomeScreen.tsx`,
     ultimately from `App.tsx`) but **never calls it anywhere** — it's dead code sitting next to
     a real, working `AddSearchScreen`. `design/screens2.jsx`'s `ListScreen` appbar already has a
     `{Ic.search}` `iconbtn` next to the back button (line ~34) that this app's `ListScreen.tsx`
     appbar is missing. Add it: a search `iconbtn` in `ListScreen.tsx`'s `.appbar`, between the
     title and the sync badge, calling `onOpenAddSearch`. This gives every open list **one**
     consistent, always-visible way to reach the dedicated Add/Search surface (matching the
     bottom-nav "Добавяне" tab's destination), without touching the existing inline
     term/quantity/unit quick-add bar or its tests.
