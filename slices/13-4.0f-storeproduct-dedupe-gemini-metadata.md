SLICE §4.0f — StoreProduct dedupe across users + async Gemini metadata extraction

BUDGET
  Target: large slice, budget 2-3x a normal slice (~250-350 changed lines,
  including the new HttpClientInterface::post() method, CLI script, and
  admin settings page). If genuinely stuck on the Gemini integration itself
  (e.g. exact response parsing), ship the dedupe half clean and flag the
  extraction half as partial per escalation §5 type 3 — don't let one stuck
  sub-part block the whole slice.

GOAL
  (1) StoreProductService::findOrCreate dedupes by normalized_name across
  ALL users, not just the creating one — so two different accounts typing
  the same specific item (e.g. "Мляко Олимпус 2% 1л") share one row instead
  of creating duplicates. (2) After a StoreProduct is created, a background
  job calls the Gemini API to extract brand/size/variant/category metadata
  from the free-text name and writes it — asynchronously, never blocking
  the add-to-list path. (3) A new admin settings page for the Gemini API
  key + model, mirroring the existing Google Sign-In settings page.

IMPLEMENTATION NOTES (as built, 2026-06-19, by Claude directly — not Codex,
  per explicit Owner instruction overriding the normal execution-model
  split for this one slice)

  - `StoreProductRepositoryInterface` gained `findByNormalizedName`,
    `findPendingMetadataExtraction(int $limit)`, `updateMetadata(...)`,
    `attachCategory(int, int)`. `StoreProductService::findOrCreate` calls
    `findByNormalizedName` before insert — exact match across all users,
    first writer wins, no merge.
  - Migration id `8` (`AddStoreProductMetadataExtractionMigration`) adds
    `store_products.size_text`, `variant_text`, `metadata_extracted_at`
    (all nullable). Registered in `MigrationRegistry::defaultMigrations()`.
  - `Models/StoreProduct` gained `sizeText`/`variantText`/
    `metadataExtractedAt` as trailing optional constructor params (default
    null) so every existing positional `new StoreProduct(...)` call site
    across the test suite kept working unmodified.
  - `HttpClientInterface::post()` added; `WpHttpClient::post()` uses
    `wp_remote_post()`, mirrors `get()`'s is_wp_error/status/body handling,
    deliberately does NOT run the blocked-status/challenge-page detection
    `get()` has (Gemini calls don't need it).
  - `Models/GeminiExtractionResult` (brand/size/variant/category, all
    nullable). `Services/GeminiMetadataExtractor::extract()` calls
    Gemini's `generateContent` REST endpoint with a JSON-only prompt,
    strips a possible ` ```json ` fence, parses the result; any HTTP
    failure, non-2xx, or unparseable JSON returns an all-null DTO rather
    than throwing (never crashes the background job).
  - `Services/StoreProductMetadataService::processPending(int $limit = 20)`
    pulls pending rows oldest-first, calls the extractor, writes via
    `updateMetadata`, and on a non-null extracted category does a
    case-insensitive name/slug match against `CategoryRepositoryInterface
    ::listAll()` — attaches via `attachCategory` on match, silently skips
    on no match.
  - `bin/extract-metadata.php` mirrors `bin/crawl.php`'s bootstrap shape
    exactly (same wp-load.php probe, same autoloader registration), wires
    `WpHttpClient`/`Clock`/`Logger`/`Config`, calls `processPending()`
    once, exits. Not wired to `wp_schedule_event` — same operational
    pattern as `bin/crawl.php` (real crontab, not WP-Cron).
  - `ConfigInterface`/`Config` gained `geminiApiKey()`/`geminiModel()`
    (`get_option('si_gemini_api_key', '')` / `get_option('si_gemini_model',
    'gemini-1.5-flash')`), mirroring `googleClientId()`/`googleClientSecret()`.
  - `Admin/GeminiSettingsPage.php` is a near-exact copy of
    `GoogleSettingsPage.php`: Settings → SI Gemini API, API key (password
    input) + model (text input, default `gemini-1.5-flash`). Wired into
    `Plugin.php` exactly where `GoogleSettingsPage` is wired (new
    `self::$geminiSettingsPage` static, same `services()`/`bootstrap()`
    pattern).
  - PHPUnit `SqliteWpdb` test stub: added the three new nullable columns
    to `store_products`. Also fixed a latent stub bug this slice's tests
    exposed — `WpdbNullSafe::bind()`'s pattern of pre-quoting a nullable
    string value and interpolating the already-quoted SQL literal into an
    outer `$wpdb->prepare()` call works in real WordPress (core `prepare()`
    escapes literal `%` before its own `vsprintf`), but the test stub's
    naive `vsprintf`-based reimplementation didn't replicate that escaping
    — so a value containing a literal `%` (e.g. `variant_text = "2%"`)
    crashed `vsprintf` with "Unknown format specifier". Fixed in the stub
    only (`tests/Repositories/SqliteWpdb.php::prepare()`), not in
    production code — this was a test-double gap, not a real bug; every
    other `WpdbNullSafe::bind()` call site in the codebase had simply
    never been exercised with a literal `%` in its bound value before.
  - Two `HttpClientInterface` anonymous-class test doubles in
    `tests/CrawlerTest.php` needed a throwing `post()` stub added since
    the interface gained a new required method.
  - New tests: `StoreProductServiceTest` (cross-user dedupe returns same
    id; different wording still creates a new row), `WpdbStoreProductRepositoryTest`
    (`findByNormalizedName` round-trip, `findPendingMetadataExtraction`
    oldest-first + limit + excludes already-extracted, `updateMetadata`
    field+timestamp write, `attachCategory` junction round-trip),
    `GeminiMetadataExtractorTest` (well-formed parse, markdown-fenced
    parse, malformed-JSON-returns-nulls, non-2xx-returns-nulls,
    client-throws-returns-nulls), `StoreProductMetadataServiceTest`
    (writes fields + attaches matching category, skips attachment on no
    category match, returns 0 when nothing pending). All 137 PHPUnit tests
    pass (was 122 before this slice).
