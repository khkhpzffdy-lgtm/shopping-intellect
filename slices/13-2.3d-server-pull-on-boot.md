BUDGET
  Target: one Codex session, under ~45 min, under ~250 changed lines (excluding test
  file diffs). This is a **correctness fix for a real data-visibility bug**, not a new
  feature ‚Äî the endpoints already exist and are already implemented server-side
  (`plugin/src/Api/ListController.php`); this slice only adds the client-side fetch +
  merge that was always supposed to exist per `07-frontend.md` ¬ß3.1 but never got

  or adding a new merge-conflict UI, STOP and hand back per escalation ¬ß5 (type 3).

GOAL
  **The bug (found 2026-06-17, Owner-reported):** logging into the same account from a
  second browser/incognito session shows **zero** of the lists created in the first
  session. This is not a cache issue ‚Äî it is a missing feature. `app/src/storage/db.ts`'s
  `getLists()`/`getListItems()` read **only** from local IndexedDB, which is populated
  exclusively by local writes. Nothing in the app ever calls `GET /lists` or
  `GET /lists/{id}` to pull what the *server* knows about the account. Two browsers, same
  account, same server-side data ‚Äî but each browser only ever sees what it personally
  wrote.

  `07-frontend.md` ¬ß3.1 already specifies the intended behaviour: "data hydrates from
  IndexedDB first (¬ß4), then **reconciles with the network**." The IndexedDB-first half
  is built (¬ß2.2b/¬ß2.2c/¬ß4.0); the network-reconciliation half was never implemented.
  This slice builds it.

  Owner-visible outcome: log in with the same account on any device/browser and see the
  **same lists and items** ‚Äî created on this device or any other ‚Äî not just the ones
  this specific browser happened to write locally.

SCOPE ‚Äî one fetch-and-merge pass, wired into the existing boot flow

  1. **Add `GET /lists` and `GET /lists/{id}` calls to `app/src/api/client.ts`.**
     Both endpoints already exist and are authenticated/working server-side
     (`plugin/src/Api/ListController.php` lines 40-42, `handleList`/`handleGet`). Add
     thin typed wrappers (e.g. `fetchLists(): Promise<ServerListDto[]>` and
     `fetchListWithItems(id): Promise<ServerListWithItemsDto>`) following the existing
     `apiRequest` pattern already used elsewhere in this file ‚Äî same error handling,
     same `authenticated: true` convention.

  2. **Add a merge function to `app/src/storage/db.ts`** ‚Äî e.g.
     `mergeServerList(serverList)` / `mergeServerListItem(serverItem, listKey)` ‚Äî that
     reconciles one server-side record into the local IndexedDB mirror:
     - If no local record exists for that server `id`, write a new local record (using
       the server-issued `client_uuid` if the server returns one for that row, or
       treating the server `id` as the lookup key ‚Äî check `04-database.md` for whether
       `lists`/`list_items` rows carry their originating `client_uuid` back in the GET
       response; if they don't yet, that's a backend gap to flag, not something to
       invent a client-side workaround for).
     - If a local record **with the same server `id`** already exists, apply
       **last-write-wins on `updated_at`** per `07-frontend.md` ¬ß4.3 ‚Äî compare
       timestamps, keep whichever is newer. Never overwrite a local record that has a
       **pending or failed** mutation queued against it (`getMutationStatusCounts`) ‚Äî
       an unsynced local edit must win over a stale server read until it actually
       syncs, or you'll silently drop the user's own in-flight edit.
     - This is a merge, not a replace: never delete a local list/item just because the
       server's `GET /lists` response doesn't currently include it in this slice (a
       list could be mid-creation and not yet visible server-side). Deletion
       reconciliation (server-side delete syncing back down) is explicitly out of
       scope below.

  3. **Wire the pull into `app/src/components/HomeScreen.tsx`'s existing boot
     `useEffect`** (the one that currently only calls `refreshLists()` on mount,
     ~line 94-96): after the existing local-first `refreshLists()` call (so the shell
     still paints instantly from cache per `07` ¬ß3.1), fire the `GET /lists` fetch, run
     each result through `mergeServerList`, then call `refreshLists()` again so the UI
     picks up anything newly merged. Do the same per-list-items fetch when a list is
     opened (the existing `refreshItems` `useEffect` keyed on `selectedListKey`,
     ~line 98-104) ‚Äî fetch `GET /lists/{id}` and merge its items in before/alongside
     the local-first read.
     - **Offline boot must not block or error.** If the `GET /lists` call fails
       (network down, matching `07-frontend.md` ¬ß3.3's "offline boot" case), swallow it
       silently and keep showing the local-first data ‚Äî exactly like the existing
       `runQueuedMutationDrain`'s `try/finally` pattern in this same file already does
       for the mutation flush. Do not add a new user-facing error message for this.
     - This pull runs **once per boot** (and once per list-open), not on a poll/interval
       ‚Äî no new periodic sync timer in this slice.

  4. **A list/item with only a server `id` and no local `client_uuid` yet must still
     render correctly** in `ListsScreen`/`ListScreen` ‚Äî check `ShoppingListRecord`/
     `ListItemRecord`'s `client_uuid` field is genuinely optional end-to-end (it's typed
     optional in `db.ts` already for the *id* field, but `client_uuid` is currently the
     `keyPath` for the IndexedDB object store, i.e. assumed always present). If the
     server doesn't hand back a `client_uuid` for rows it created from another device's
     sync, decide (and document inline with a one-line comment, not a essay) whether
     this slice generates a fresh local `client_uuid` for bookkeeping or whether the
     backend needs to start returning the originating `client_uuid` in `GET` responses
     ‚Äî check `04-database.md` for whether `lists`/`list_items` tables even store
     `client_uuid` server-side (CLAUDE.md Iron Rule 6 says they do). If they do, this is
     a one-line addition to the existing `handleList`/`handleGet` PHP response shape,
     not a new column.

OUT OF SCOPE (explicitly ‚Äî do not pull these in)
  - **Server-side delete propagation.** If a list is deleted on one device, this slice
    does not need to make it disappear on another device's next boot ‚Äî that requires a
    tombstone/deleted-list mechanism that doesn't exist yet (no `DELETE /lists/{id}`
    endpoint exists server-side either ‚Äî check `06-api-auth.md`). Flag it as a follow-up,
    don't build it now.
  - **Real-time sync / websockets / polling.** This is a boot-time + list-open pull,
    not a live multi-device sync. Two browsers open simultaneously will not see each
    other's edits until the next boot/list-open.
  - **Family-shared lists' merge semantics beyond what `GET /lists` already returns.**
    `06-api-auth.md` says `GET /lists` returns "all lists the caller owns or can see
    (personal + every family)" ‚Äî trust that response as-is; don't add client-side
    filtering logic.
  - **A conflict-resolution UI.** Last-write-wins, silently, per `07-frontend.md` ¬ß4.3 ‚Äî
    no merge dialog, no "keep mine / keep theirs" prompt.
  - **Changing `POST`/`PATCH`/`DELETE` mutation behavior** ‚Äî this slice is additive
    (a new read path); the existing write/queue pipeline (`sendMutation`, `flush.ts`)
    is untouched.
  - **`GET /lists/{id}/comparison` or any pricing/offer data** ‚Äî unrelated read path,
    not touched here.
